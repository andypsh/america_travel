4. Git 브랜치 전략 (Gitflow-lite)

혼자 작업이지만 브랜치를 3단으로 나눠서 "프로덕션 / 통합 / 작업" 분리.

main ────●────●────●────● ← Vercel이 이 브랜치를 배포
▲
│ fast-forward merge
dev ─────●────●────●────● ← 통합·QA
▲
│ fast-forward merge
feature_andy ●────●────●────● ← 현재 작업 (커밋 진행 중)
브랜치역할누가 푸시
main프로덕션 · Vercel 자동 배포 대상안정화 후 dev에서 fast-forward
dev통합 · 풍부한 상태(3탭 등) 보관 가능feature_andy에서 fast-forward
feature_andy현재 WIP — 모든 새 커밋매 작업 단위

왜 fast-forward만?

머지 커밋(merge commit) 없이 선형 히스토리 유지 → git log --oneline 이 깔끔. 이력 추적 쉬움. 작은 프로젝트엔 충분.

# feature_andy에서 작업 후
git checkout dev
git merge --ff-only feature_andy   # dev = feature_andy
git push origin dev

git checkout main
git merge --ff-only feature_andy   # main = feature_andy
git push origin main               # ← Vercel 자동 배포 트리거

5. 서버 배포 101 — "이거 잘 모르겠는데?"

웹사이트를 인터넷에 띄우는 방식은 크게 둘로 나뉜다.

🟢 정적 사이트 (Static)

미리 빌드된 HTML/JS/CSS 파일을 그대로 CDN에 올림. 서버는 그냥 파일 전달만.

  • 예: 블로그·랜딩페이지·이 프로젝트
  • 장점: 빠름, 싸움, 보안 강함
  • 한계: 사용자별 다른 데이터 불가능 (DB 직접 못 씀)

🟡 동적 사이트 (Dynamic / 서버 렌더)

서버에서 요청마다 HTML 생성. Node·Python·Go 같은 런타임 24/7 돌림.

  • 예: 인스타그램·쿠팡·은행 사이트
  • 장점: 사용자별 맞춤·실시간 데이터
  • 비용: 서버 비용·DB 비용·운영 관리
📍 이 프로젝트는 정적. 일정 데이터는 코드에 박혀있고, 사용자별 입력도 없음. 그래서 Vercel 정적 호스팅으로 충분 + 무료.

6. DB 구성 101 — "데이터베이스가 뭐길래?"

웹사이트가 사용자 정보·게시글·주문 같은 걸 기억해야 할 때 DB가 필요. 코드(JS·Python)는 메모리에 있다가 서버 재시작하면 다 사라지니까, 디스크에 영구 저장하는 별도 시스템이 DB.

🤔 그런데 이 프로젝트엔 왜 DB가 없을까?

일정 데이터(aDays, laHotels 등)는 3명만 쓰고 변동 거의 없는 상수. DB에 넣을 필요가 없어서 JS 파일에 그대로 박아뒀음. 빌드 시 번들에 인라인 → 모든 사용자가 같은 데이터를 받음.

만약 사용자가 일정을 편집해서 저장해야 한다면 그때 DB가 필요해짐 (각 사용자별 데이터 보관).

6-1. SQL vs NoSQL — 두 진영

🗃️ SQL (관계형 DB)

엑셀처럼 표(table)로 정리. 표 사이 관계(JOIN)도 강력.

  • 대표: PostgreSQL, MySQL, SQLite
  • 강함: 정합성·복잡 쿼리·트랜잭션
  • 약함: 스키마 변경 부담
  • 예: 은행 계좌·전자상거래

📦 NoSQL (비관계형)

JSON 같은 자유 형식 문서. 스키마 유연.

  • 대표: MongoDB, Firestore, DynamoDB
  • 강함: 빠른 변경·수평 확장
  • 약함: 복잡한 관계 쿼리
  • 예: 로그·실시간 채팅·IoT
⚖️ 안 모르겠으면 PostgreSQL부터 시작. SQL은 어디서나 통용되고, 나중에 옮기기도 쉬움. NoSQL은 진짜 필요할 때 (대규모 실시간 등).

8. 디버깅 회고 — 실제로 만난 함정

8-1. 빈 화면 (White Screen of Death)

증상: npm run dev 후 페이지 빈 화면. 콘솔에 모듈 로드 실패.

원인: index.html 이 /src/main.js 참조 → 실제 파일은 루트의 main.js. Vite가 못 찾음.

해결:

- <script type="module" src="/src/main.js"></script>
+ <script type="module" src="/main.js"></script>

8-2. Vite 프록시 광범위 매칭 (가장 헷갈렸음)

증상: 페이지 여전히 빈 화면. 콘솔에 /api.js 에 대해 ECONNREFUSED 에러.

원인: vite.config.js 의 proxy 패턴이 '/api' 로 시작하는 모든 경로를 매칭. App.vue가 ./api.js import 하면 /api.js 가 되는데 → 죽은 백엔드(:8000)로 가로채짐 → 모듈 로드 실패.

해결: 정규식 prefix 사용

// before
proxy: { '/api': { target: '...' } }

// after — '/api/' 슬래시 포함만
proxy: { '^/api/': { target: '...' } }
💡 교훈: Vite proxy 패턴은 prefix 매칭 → 짧으면 광범위. 특히 /api는 위험.

8-3. Node 16 + Vite 5 비호환

증상: crypto$2.getRandomValues is not a function 에러로 dev 서버 못 띄움.

원인: Vite 5는 Node 18+ 필요. 로컬 Node가 16.

해결: package.json 에서 Vite 4.5로 다운그레이드 (Node 14.18+ 지원)

{
  "devDependencies": {
    "@vitejs/plugin-vue": "^4.6.0",  // 5 → 4
    "vite": "^4.5.0"                  // 5 → 4
  }
}

8-4. 다른 워크트리에서 푸시된 커밋 보존

상황: 다른 작업 세션이 main 에 작은 패치 직접 푸시. 내 feature_andy 와 분기.

해결: 해당 커밋만 cherry-pick으로 가져와서 feature_andy에 통합. 그 후 main도 강제 정렬.

git fetch origin
git cherry-pick <sha>            # 다른 세션 커밋만 가져오기
git push origin feature_andy
# main/dev도 같은 시작점으로 정렬

9. 용어 사전 (자주 헷갈리는 것들)

용어한 줄 설명
SPA (Single Page App)HTML 하나로 시작해서 JS가 화면을 다시 그림. 페이지 이동 없음. 이 프로젝트가 그것.
CDN전 세계 분산된 캐시 서버. 사용자에게 가까운 곳에서 파일 받게 함. 빠름.
EdgeCDN의 한 노드 (서울/도쿄/싱가포르 등). "엣지에서 실행"한다고 하면 사용자 가까이서 코드 실행 의미.
HMR (Hot Module Replacement)코드 저장 시 페이지 새로고침 없이 변경 부분만 교체. Vite/Webpack의 dev 기능.
Webhook이벤트 발생 시 다른 서버에 자동 알림 보내는 방식. GitHub push → Vercel 알림이 webhook.
서버리스 함수24/7 안 돌고 요청 올 때만 실행되는 함수. 안 쓸 때 0원. 콜드스타트(첫 실행 지연) 있음.

커넥션 풀DB 연결을 미리 여러 개 만들어 두고 재사용. 매번 새로 만들면 느림.
